# claude-discord-bridge

A small daemon that mirrors [Claude Code](https://claude.com/claude-code) sessions into a Discord channel and lets you reply from anywhere — including your phone — by piping replies back into the originating `tmux` pane.

```
Claude Code hook ──► notify.sh ──HTTP──► bot.py ──► Discord channel/thread
                                                          ▲
                                                          │ your reply
                                                          ▼
                                  tmux paste-buffer + Enter
```

Each project / branch / agent / tmux pane gets its own Discord thread, with the last assistant text attached on `Stop` events. Image attachments you send in Discord are downloaded to a local inbox and injected as `@<path>` references that Claude Code reads natively.

## Why

You start a long task in Claude Code, walk away, want to know when it asks a question — and answer it without coming back to the laptop. This does that, and only that.

## Requirements

- Linux (tested on Fedora). Should work on any system with `tmux`, `python 3.11+`, `jq`, `curl`.
- A Discord application + bot token with **Message Content Intent** enabled.
- A Discord channel where the bot can post + create threads.
- Claude Code running inside `tmux` (the wrapper script and hook can pass through pane info even when `$TMUX_PANE` is stripped).

## Install

```bash
git clone https://github.com/mrw1986/claude-discord-bridge.git ~/claude-discord-bridge
cd ~/claude-discord-bridge
./setup.sh
```

`setup.sh` creates the Python venv, generates a `BRIDGE_TOKEN`, links the systemd user unit, and prints what to fill in next.

### `.env`

Fill these in `~/claude-discord-bridge/.env`:

| Var | Required | Description |
| --- | --- | --- |
| `DISCORD_BOT_TOKEN` | yes | Bot token from the Discord developer portal. |
| `DISCORD_CHANNEL_ID` | yes | Channel where threads are created. |
| `DISCORD_ALLOWED_USER_IDS` | yes | Comma-separated user IDs allowed to reply (anyone else is rejected — see *Security*). |
| `BRIDGE_TOKEN` | yes | Shared secret between `notify.sh` and `bot.py` (auto-generated by `setup.sh`). |
| `BRIDGE_PORT` | no | Local HTTP port. Default `7777`. |
| `DISCORD_PING_USER_ID` | no | If set, `notification` events `@`-mention this user (needed for mobile push, since bot posts in unjoined threads don't push by default). |

### Start the bridge

```bash
systemctl --user daemon-reload
systemctl --user enable --now claude-discord-bridge.service
journalctl --user -u claude-discord-bridge.service -f
```

You should see `Bot ready: <bot>#<discriminator>` and `HTTP listening on 127.0.0.1:7777`.

### Wire up Claude Code hooks

Add to `~/.claude/settings.json`:

```jsonc
{
  "hooks": {
    "Notification": [
      { "matcher": "", "hooks": [
        { "type": "command", "command": "/home/<you>/claude-discord-bridge/notify.sh notification", "timeout": 5, "async": true }
      ] }
    ],
    "Stop": [
      { "matcher": "", "hooks": [
        { "type": "command", "command": "/home/<you>/claude-discord-bridge/notify.sh stop", "timeout": 5, "async": true }
      ] }
    ],
    "SubagentStop": [
      { "matcher": "", "hooks": [
        { "type": "command", "command": "/home/<you>/claude-discord-bridge/notify.sh subagent_stop", "timeout": 5, "async": true }
      ] }
    ]
  }
}
```

`notify.sh` is fail-soft — if the bridge is down, `jq` is missing, or the token isn't set, it logs to stderr and exits `0` so Claude Code is never blocked.

## How it works

### Outbound (Claude → Discord)

1. A Claude Code hook calls `notify.sh <event>`.
2. `notify.sh` reads the hook JSON from stdin, captures `project`, `branch`, `agent`, current `tmux_pane` (+ `pane_pid`), and the `transcript_path`.
3. It `POST`s to `http://127.0.0.1:$BRIDGE_PORT/notify` with `X-Bridge-Token`.
4. `bot.py` finds or creates a thread keyed by `(project, branch, agent, pane)`, posts the event, and on `stop` / `subagent_stop` attaches the last assistant message either inline or as a `transcript.txt` file.

### Inbound (Discord → Claude)

1. You reply in the thread.
2. `bot.py` checks your Discord user ID against `DISCORD_ALLOWED_USER_IDS`.
3. The text (plus any image attachments saved to `~/.local/share/claude-discord-bridge/inbox/<thread>/`) is shoved into the pane via `tmux load-buffer` + `paste-buffer` + `send-keys Enter`.
4. Image attachments are referenced with Claude Code's `@<path>` syntax so they go in as multimodal input.

### Pane self-healing

The stored `tmux_pane` (e.g. `%0@claude-bridge`) can go stale across tmux restarts. The bot also stores `pane_pid`. At send time it verifies the pane still exists; if not, it walks `list-panes -a` across known sockets matching by PID and updates the mapping. Replies don't silently land in the wrong pane.

## Security

This is a **remote shell** into your Claude Code session. Take it seriously.

- `/notify` requires `X-Bridge-Token`. Treat it like a password.
- Discord reply → `tmux send-keys` only fires for IDs in `DISCORD_ALLOWED_USER_IDS`. **Anyone on that list can effectively type commands into your Claude session.** Keep the list to people you trust at host-shell level.
- Threads are public (within the channel). Don't post secrets, and assume anyone with channel read access can see project names, branches, and any `Stop`-event transcript snippets.
- The systemd unit runs as your user without sandboxing — same blast radius as your normal shell. Consider `NoNewPrivileges=true`, `PrivateTmp=true`, and an explicit `ReadWritePaths` if you want to harden.
- `notify.sh` sources `.env` with `set -a; . .env; set +a`. Keep `.env` mode `600` (setup does this) and don't let anything untrusted write it.

A more thorough adversarial review (path traversal, MIME spoofing, supply chain, stale-pane redirection, etc.) lives in the project notes — issues / PRs welcome if you want to land more of the hardening.

## Project layout

```
bot.py                          aiohttp HTTP + discord.py gateway
notify.sh                       Claude Code hook entrypoint
setup.sh                        Bootstrap: venv, .env, systemd link
claude-discord-bridge.service   systemd user unit (EnvironmentFile=./.env)
requirements.txt                discord.py, aiohttp
```

State lives at `~/.local/state/claude-discord-bridge/threads.json` (atomic writes). Image inbox at `~/.local/share/claude-discord-bridge/inbox/`.
